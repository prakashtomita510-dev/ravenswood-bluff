# Alpha 0.2 专项计划：全部角色业务实现与验证

## 背景

`alpha 0.1` 已经完成了相当一批高风险角色与主链角色的实现和回归，但还没有达到“Trouble Brewing 全角色业务实现与规则验证完整闭环”的程度。

当前项目已经具备：

- 角色定义框架
- 规则矩阵
- 若干高风险角色回归
- 基础昼夜流程与死亡触发能力

接下来需要把“角色是否存在”推进到“角色是否完整可用、是否与规则书一致、是否已被自动验证”。

---

## 总体目标

围绕 `Trouble Brewing / 暗流涌动`，建立：

1. 全角色业务实现清单
2. 全角色规则符合性清单
3. 全角色自动化验证清单

## 二、角色实现的统一模板

每个角色都应按同一模板审视：

1. 触发时机是否正确
2. 输入输出是否正确
3. 是否接入说书人裁定链
4. 是否考虑醉酒/中毒/误判等扰动
5. 是否与昼夜流程和死亡流程正确挂接
6. 是否有自动回归测试

---

## 三、重点风险类别

### 1. 固定信息角色

重点风险：

- 信息格式不统一
- 假信息扰动不统一
- 时机不稳定

### 2. 夜晚主动目标角色

重点风险：

- 目标合法性
- 动作顺序
- 保护/免疫/中毒交互

### 3. 被动/触发类角色

重点风险：

- 圣女首提触发
- 圣徒败北触发
- 守鸦人夜死触发
- 市长夜死转移
- 绯红女郎接管

### 4. setup 影响型角色

重点风险：

- 男爵外来者增量
- 角色发牌分布
- 酒鬼假身份选择

## 五、验证体系

### 1. 单角色规则测试

每个角色至少需要：

- 1 个基础规则正确性测试
- 复杂角色 2 到 4 个边界测试

### 2. 组合交互测试

测试重点：

- 投毒者 + 信息位
- 僧侣 + 小恶魔
- 隐士 / 间谍 + 侦测角色
- 圣女 / 圣徒 + 处决链
- 绯红女郎 + 恶魔死亡

### 3. 端到端剧本测试

建议建立几种固定剧本：

- 信息位主导局
- 邪恶干扰局
- 圣女 / 圣徒边界局
- 夜死触发局

## 七、验收标准

视为完成 `alpha 0.2` 角色专项的标准：

- Trouble Brewing 全角色都有明确实现状态
- 所有高风险角色都有自动回归
- 关键角色交互有组合测试
- 说书人裁定链和角色逻辑不再分叉
- 规则矩阵覆盖完整、可追踪

---

## 结论

角色专项的重点不是“把角色类都写出来”，而是：

- 让每个角色真的按规则工作
- 让角色之间能正确交互
- 让每个 bug 都能通过测试和矩阵追溯

这会是 `alpha 0.2` 规则一致性收口的核心基础。
