# Alpha 0.2 专项计划：前端界面优化

## 背景

`alpha 0.1` 的前端已经从早期原型提升到了“可玩、可操作、可显示关键状态”的阶段，但整体仍以功能正确性为优先，体验层还有明显打磨空间。

从实测反馈来看，前端目前最容易出现的问题有：

- 阶段切换时状态不够直观
- 关键私密信息容易与公共信息混淆
- 提名/投票/处决链的信息层级不够清晰
- 玩家端和说书人端的信息密度、布局方式还不够理想

因此，`alpha 0.2` 需要把前端界面优化单独拉成专项推进。

---

## 总体目标

让前端从“能显示和能操作”，提升为：

1. 清晰
2. 沉浸
3. 可回溯
4. 适合真实对局决策

---

## 一、玩家端优化

### 核心目标

玩家端应该优先服务于“社交推理”和“即时决策”，而不是把所有状态都堆在一个页面里。

### 优化方向

#### 1. 聊天室与状态页分层

继续优化两页结构：

- `聊天室`
- `状态页`

状态页内部建议进一步细分：

- 当前提名状态
- 历史提名记录
- 私密信息归档
- 当前死亡与公开事件

#### 2. 提名/投票/处决可视化

提升显示：

- 当前阶段
- 当前轮次
- 提名者 / 被提名者
- 当前票数
- 已投票玩家
- 历史提名记录
- 最终处决结果

#### 3. 私密信息优化

私密信息面板要进一步强调：

- 仅自己可见
- 分夜晚分组
- 支持历史回看
- 不与公共日志混在一起

#### 4. 死亡与状态变化提示

所有玩家都应清楚看到：

- 谁死了
- 因为什么死
- 当前还剩多少存活玩家

### 验收标准

- 玩家能够不切换页面也理解当前局势
- 状态页不会遮挡关键聊天信息
- 提名链和死亡链清楚可见

---

## 二、说书人端优化

### 核心目标

说书人端应服务于“主持整局游戏”，而不是只做一个普通控制台。

### 优化方向

#### 1. 夜晚步骤工作台

说书人页面要清楚展示：

- 当前夜晚步骤
- 下一位被唤醒的玩家
- 当前可执行动作
- 候选信息

#### 2. 魔典展示增强

增强魔典表现：

- 真实身份
- 感知身份
- 当前队伍
- 死亡状态
- 中毒 / 醉酒 / 保护
- 关键标记

#### 3. 说书人日志摘要

支持直接在页面里查看：

- 最近裁定
- 最近私密信息发放
- 最近状态变化

### 验收标准

- 人类说书人无需翻日志文件就能主持夜晚
- 魔典信息足够完整且可读

---

## 三、视觉与信息结构优化

### 当前问题

当前界面已经能用，但信息层级仍然偏平，容易让玩家在关键阶段漏看信息。

### 改进方向

- 强化阶段标题
- 关键事件用颜色或图标强调
- 当前操作区域更明确
- 历史区与当前区分层
- 桌面端与移动端都保持可读

### 原则

- 优先清晰度，不追求花哨
- 不遮挡核心决策信息
- 不让状态区抢占聊天室主导地位

---

## 四、前端状态管理优化

### 目标

继续压缩“旧状态残留”“局间污染”“事件和轮询互相打架”的问题。

### 重点方向

- 按 `game_id` 隔离状态
- 当前状态与历史状态分离
- websocket 事件与轮询 state 的统一合并策略
- 关键 UI 状态显式清理

### 验收标准

- 新开一局时前端状态完全干净
- 提名状态结束后立即切换，不残留旧阶段
- 游戏结束后状态页不显示误导性旧内容

---

## 五、分阶段实施建议

### Wave 1：信息结构与状态清理

- 提名状态页优化
- 私密信息优化
- 死亡播报优化
- `game_id` 隔离与状态清理

### Wave 2：说书人工作台增强

- 夜晚步骤
- 魔典增强
- 裁定摘要

### Wave 3：视觉与体验打磨

- 信息层次优化
- 可读性提升
- 响应式布局与微交互

---

## 六、验收方式

前端优化必须经过：

- 静态检查
- API 契约检查
- 真人手动试玩
- 前端自动化验收

---

## 结论

前端优化的目标不是“更漂亮”，而是：

- 更利于决策
- 更利于主持
- 更不容易出错

它应当服务于《血染钟楼》的核心体验：社交推理与说书人主持。
