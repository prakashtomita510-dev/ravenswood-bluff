# Alpha 0.2 专项计划：自动化验收与测试

## 背景

`alpha 0.1` 已经建立了一批测试资产，包括：

- 单元测试
- 规则测试
- API 契约测试
- 前端契约测试
- 模拟局测试
- 多轮真人前端实测

但进入 `alpha 0.2` 后，系统复杂度会继续提高：

- 说书人裁定更复杂
- AI 玩家人格与记忆更复杂
- 前端状态更复杂
- 角色交互更多

因此，测试体系必须从“有测试”升级到“有分层验收体系”。

---

## 总体目标

建立覆盖以下四层的验收体系：

1. 后端规则验收
2. API 契约验收
3. 模拟对局验收
4. 真人前端自动化验收

---

## 一、后端规则验收

### 目标

确保：

- 角色规则正确
- 昼夜流程正确
- 事件顺序正确
- 胜负判定正确

### 测试类型

- 单角色规则测试
- 角色组合交互测试
- 昼夜流程测试
- 提名/投票/处决测试
- 死亡与胜负测试

### 重点用例

- 圣女 / 圣徒
- 守鸦人
- 小恶魔 / 僧侣 / 市长
- 投毒者 + 信息位
- 绯红女郎接管

### 验收标准

- 每个高风险角色至少有对应规则回归
- 所有关键流程都有顺序性断言

---

## 二、API 契约验收

### 目标

确保前端和说书人端依赖的 API 输出稳定，不因内部实现变化而悄悄破坏界面。

### 范围

- `/api/game/state`
- `/api/game/metrics`
- `/api/game/grimoire`
- `/api/game/setup`
- `/api/game/reset`
- `/api/game/roles`
- `/api/storyteller/*`

### 核心检查点

- `game_id`
- `viewer_mode`
- `nomination_state`
- `private_info`
- `players`
- 魔典权限

### 验收标准

- 所有关键响应字段稳定存在
- 玩家/说书人视角严格区分
- 契约测试能覆盖最近修复过的历史问题

---

## 三、模拟对局验收

### 目标

让系统在不依赖真人参与的情况下，能够持续跑审计局和短局回归。

### 模式

- `mock`
- `live`

### 建议分层

#### 1. 快速审计局

目的：

- 检查是否能跑通
- 检查是否有提名、投票、处决
- 检查是否会卡死

#### 2. 规则压力局

目的：

- 检查多角色组合
- 检查多轮提名
- 检查夜晚链稳定性

#### 3. 智能行为观测局

目的：

- 观察 AI 是否过于机械
- 提名与投票是否合理
- 阵营行为是否有差异

#### 4. 说书人裁量评估局

目的：

- 记录一整局上下文中的裁量节点
- 观察说书人是否在规则边界内做出合理裁量
- 评估游戏是否因裁量而过早锁死或过早结束

建议输出：

- 完整 truth view
- 裁量候选与最终选择
- 当前善恶人数与局势热度
- 关键私密信息发放历史
- 是否出现单边碾压或 hard lock 风险

### 验收标准

- mock 局稳定跑通
- live 局至少能跑完核心白天链
- 关键指标可观测

---

## 四、真人前端自动化验收

### 目标

把真人前端实测中最容易反复出问题的链路，沉淀成可复用的自动化验收。

### 覆盖范围

- 连接
- setup
- 首夜私密信息
- 聊天室/状态页切换
- 提名
- 辩解
- 投票
- 处决
- 死亡播报
- 魔典权限
- 新局隔离

### 推荐方式

- Playwright MCP
- 页面状态快照
- 关键 API 辅助断言

### 建议固定脚本化场景

1. 人类玩家整局短局
2. 人类说书人夜晚流程
3. 开两局验证前端状态隔离
4. 守鸦人 / 私密信息 / 死亡播报专项场景

### 验收标准

- 至少有一套可重复执行的前端自动化验收脚本
- 关键历史 bug 能通过自动化复现并防回归

---

## 五、可观测性与调试支撑

### 目标

让测试失败时能快速定位。

### 需要持续完善的输出

- `orchestrator_run.log`
- `storyteller_run.log`
- `metrics`
- 结构化事件日志
- AI 决策日志

### 建议增加的指标

- 提名成功率
- 投票通过率
- 死亡触发成功率
- 私密信息投递成功率
- 前端会话污染复现率
- 过早结束率
- `final 3` / `final 4` 到达率
- 说书人自由裁量次数
- 裁量后局势失衡率
- `hard lock` 触发率

---

## 六、回归分层建议

### PR 级

- 单元测试
- API 契约测试

### 日常开发级

- 关键流程回归
- mock 模拟局

### 版本级

- 全量测试
- 多组模拟局
- 真人前端自动化验收
- 关键手工试玩

## 结论

测试体系不是附属物，而是 `alpha 0.2` 能否稳定推进的基础设施。

随着项目复杂度提升，必须形成：

- 后端规则验收
- API 契约验收
- 模拟局验收
- 真人前端自动化验收

四层联动的完整体系，才能保证后续每一轮增强都不会把旧问题重新带回来。
