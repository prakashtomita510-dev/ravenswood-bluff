# Alpha 0.2 路线图

## 目标

这份路线图用于把 `alpha 0.2` 的开发顺序从“方向”进一步细化为“可执行阶段”。

核心原则是：

1. 先稳规则主链
2. 再稳说书人真相源
3. 再补角色完整性
4. 再增强 AI 智能
5. 再打磨前端体验
6. 最后用自动化验收收口

这样推进的原因很简单：

- 如果规则链和说书人裁定层不稳定，AI 增强只会建立在错误信息之上
- 如果前端体验先大改，而后端状态契约还在变化，界面会反复返工
- 如果没有测试门禁，后续每轮增强都容易把旧问题带回来

---

## 总体分期

建议拆成 4 个 Wave：

1. `Wave 1`：规则主链稳定化
2. `Wave 2`：说书人与角色一致性增强
3. `Wave 3`：AI 玩家智能增强
4. `Wave 4`：前端体验与封版验收

---

## Wave 1：规则主链稳定化

### 目标

确保 `alpha 0.1` 中已经打通的主流程真正稳定，不再出现影响整局可玩性的关键 bug。

### 优先事项

#### 1. 死亡触发链收口

重点：

- 守鸦人夜死触发
- 死亡后的私密信息发放
- 死亡播报公开链
- 死亡触发与夜晚结算的顺序

#### 2. 提名 / 投票 / 处决子系统稳定化

重点：

- 当前提名状态与历史提名彻底分离
- 多轮提名稳定
- 当前状态结束后即时清空
- 同票 / 无人提名 / 无人处决边界
- AI 与人类玩家的提名窗口一致

#### 3. 前端局间状态污染彻底清理

重点：

- `game_id` 驱动的会话隔离
- 聊天记录不串局
- 私密信息不串局
- 提名状态不串局
- websocket 与轮询一致性

#### 4. 固定信息与夜晚链路补缺

重点：

- 共情者
- 调查员
- 邪恶首夜互认
- 间谍魔典刷新
- 夜晚私密信息显示链

### 依赖关系

无前置依赖，直接开始。

### 交付物

- 规则主链 bug 清单关闭
- 关键流程回归全绿
- 真人短局可稳定打完

### 验收门槛

- 模拟局稳定跑通
- 真人短局能完整经历讨论、提名、投票、夜晚、死亡播报
- 前端不再出现旧局污染

---

## Wave 2：说书人与角色一致性增强

### 目标

把项目从“能跑”推进到“说书人和角色实现基本可信”。

### 优先事项

#### 1. 说书人真相源统一

重点：

- 固定信息角色输出统一收口到说书人
- 真假信息扰动链统一
- 说书人日志增强

#### 2. 自由裁量建模

重点：

- 红鲱鱼
- 中毒 / 醉酒假信息
- 隐士 / 间谍误判
- 市长夜死转移

#### 3. 剩余高风险角色业务实现

建议优先顺序：

1. 绯红女郎
2. 男爵
3. 管家
4. 市长
5. 杀手
6. 隐士
7. 酒鬼边界

#### 4. 角色组合交互测试

重点：

- 投毒者 + 信息位
- 僧侣 + 小恶魔
- 圣女 / 圣徒
- 绯红女郎 + 恶魔死亡
- 间谍 / 隐士 + 侦测角色

### 依赖关系

依赖 `Wave 1` 基本稳定，否则角色问题和流程问题会混在一起难以定位。

### 交付物

- 说书人裁定链统一
- 剩余高风险角色实现
- 角色组合交互回归

### 验收门槛

- 规则矩阵覆盖范围扩大
- 高风险角色主链测试全部通过
- 说书人日志可解释关键信息发放

---

## Wave 3：AI 玩家智能增强

### 目标

让 AI 玩家从“规则参与者”升级为“更像真人的社交推理玩家”。

### 优先事项

#### 1. 玩家视角包重构

重点：

- 彻底隔离说书人真相层
- AI 只看玩家应知信息
- 酒鬼 / 隐士 / 间谍等特殊角色的认知边界正确

#### 2. 提名 / 投票策略去机械化

重点：

- 不再默认提名前置位
- 允许明确选择不提名
- 投票不再纯阈值化
- 区分真人局与 audit/mock 兜底行为

#### 3. Persona 扩展

重点：

- 基础人格原型库
- 行为偏置
- 角色叠加人格表现

#### 4. 记忆系统增强

重点：

- 工作记忆压缩
- 情节记忆真正接入
- 社交图谱自动更新
- 会话隔离

#### 5. 社交推理增强

重点：

- 发言结构化提取
- 怀疑度持续演化
- 带节奏 / 跟票 / 保人 / 质疑模式识别

### 依赖关系

依赖 `Wave 2`：

- 说书人裁定稳定
- 角色规则可信
- 玩家视角边界可落地

### 交付物

- AI 玩家智能增强专项完成第一版
- 提名 / 投票行为明显更自然
- 发言更像真人玩家

### 验收门槛

- 不再出现机械前置位提名
- 同局 AI 风格明显不同
- 多轮对局中能体现连续社交推理

---

## Wave 4：前端体验与封版验收

### 目标

在规则、说书人和 AI 都相对稳定后，对前端和测试体系做系统性收尾。

### 优先事项

#### 1. 玩家端体验优化

重点：

- 聊天 / 状态页信息结构优化
- 私密信息归档优化
- 死亡、提名、投票链的可视化增强
- Game Over 结算与复盘 Overlay

#### 1.5 结算与复盘系统

重点：

- 游戏结束时自动生成结算报告
- 对局记录持久化
- 历史对局查询
- Game Over Overlay
- Rematch 流程

建议将 [游戏结算与复盘系统开发计划](d:/鸦木布拉夫小镇/docs/alpha-0.2-plan/gameover_implementation_plan.md) 作为 Wave 4 的核心工作包执行。

#### 2. 说书人端工作台增强

重点：

- 夜晚步骤工作台
- 魔典增强
- 最近裁定摘要
- 风险提示

#### 3. 自动化验收建设

重点：

- 后端规则验收
- API 契约验收
- 模拟局验收
- 真人前端自动化验收

#### 4. 版本封版检查

重点：

- 高优先 bug 清零
- 测试矩阵稳定
- 文档更新
- 发布说明准备

### 依赖关系

依赖前 3 个 Wave 基本完成，否则前端和验收仍会跟着后端频繁变化。

### 交付物

- 更成熟的玩家端和说书人端
- 游戏结算、历史记录与复盘系统
- 稳定的自动化验收体系
- `alpha 0.2` 发布准备完成

### 验收门槛

- 真人前端自动化验收可重复运行
- 全量测试稳定通过
- Game Over 结算、历史记录、Rematch 流程通过专项验收
- 文档、规则矩阵和发布文档同步更新

---

## 任务拆分建议

### 适合优先并行的任务

这些任务适合并行推进：

1. 前端状态隔离与显示优化
2. 角色单体实现与角色测试
3. 说书人日志与裁定接口增强
4. 模拟局与 API 契约测试补全

### 适合串行推进的任务

这些任务建议按顺序推进：

1. 玩家视角包重构
2. AI 记忆系统增强
3. AI 提名 / 投票策略增强

原因是它们彼此依赖很强，容易互相覆盖。

---

## 每个 Wave 的收尾条件

### Wave 1 收尾

- 关键流程 bug 不再阻塞真人对局
- 旧局污染类问题清零
- 提名 / 夜晚 / 死亡链稳定

### Wave 2 收尾

- 说书人真相源基本成型
- 高风险角色主链已覆盖
- 规则矩阵明显扩展

### Wave 3 收尾

- AI 行为明显不再机械
- 人格、记忆、社交推理具备第一版可用性

### Wave 4 收尾

- 自动化验收形成体系
- 前端与文档收口
- 可进入下一阶段版本发布准备

---

## 推荐执行节奏

如果按一个比较稳妥的方式推进，我建议：

1. 先集中完成 `Wave 1`
2. `Wave 2` 可以拆成说书人线与角色线并行
3. `Wave 3` 以 AI 智能专项为主线集中推进
4. `Wave 4` 在版本接近完成时统一收尾

---

## 结论

`alpha 0.2` 不应该平均用力，而应按依赖关系推进：

先把底层流程和规则做稳，再让说书人与角色可信，再去提高 AI 智能，最后把前端体验和自动化验收收好。

这样能最大限度减少返工，也能让每一步增强都建立在稳定基础之上。
